6//! # Key Encapsulation Mechanisms (KEM)
7//!
8//! This module provides a unified interface for both post-quantum and classical
9//! key encapsulation mechanisms. It supports the following algorithms:
10//!
11//! ## Post-Quantum Algorithms
12//!
13//! - **ML-KEM (FIPS 203)**: Module-Lattice-Based Key Encapsulation Mechanism
14//!   - ML-KEM-512: NIST Security Category 1 (AES-128 equivalent)
15//!   - ML-KEM-768: NIST Security Category 3 (AES-192 equivalent)
16//!   - ML-KEM-1024: NIST Security Category 5 (AES-256 equivalent)
17//!
18//! ## Classical Algorithms
19//!
20//! - **ECDH (X25519)**: Elliptic Curve Diffie-Hellman key exchange (RFC 7748)
21//!
22//! ## Security Properties
23//!
24//! - **ML-KEM** (FIPS 203): **IND-CCA2** — indistinguishability under adaptive
25//!   chosen-ciphertext attack. This is a true KEM with an authenticated
26//!   ciphertext path; decapsulation rejects invalid ciphertexts.
27//! - **X25519 ECDH** (RFC 7748): NOT a CCA-secure KEM on its own. Raw ECDH is
28//!   a key-agreement primitive that provides **IND-CPA** (chosen-plaintext
29//!   indistinguishability) under the computational Diffie-Hellman assumption,
30//!   not IND-CCA2. Within this crate X25519 is only used as an input to the
31//!   hybrid HKDF combiner in [`hybrid::kem_hybrid`](crate::hybrid::kem_hybrid),
32//!   which upgrades the composed construction to IND-CCA2 via ML-KEM.
33//! - **Constant-time** — all secret-handling operations execute in
34//!   constant time (delegated to the backing crates `aws-lc-rs` for ML-KEM
35//!   and `x25519-dalek` for ECDH).
36//! - **Zeroization** — secret keys are scrubbed on drop.
37//!
38//! ## Example Usage
39//!
40//! ### ML-KEM Key Encapsulation
41//!
42//! ```no_run
43//! use latticearc::primitives::kem::ml_kem::{MlKem, MlKemSecurityLevel};
44//! use subtle::ConstantTimeEq;
45//!
46//! // Generate keypair
47//! let (pk, sk) = MlKem::generate_keypair(MlKemSecurityLevel::MlKem768)?;
48//!
49//! // Encapsulate shared secret
50//! let (shared_secret, ciphertext) = MlKem::encapsulate(&pk)?;
51//!
52//! // Decapsulate shared secret
53//! let recovered_secret = MlKem::decapsulate(&sk, &ciphertext)?;
54//!
55//! // Shared secrets must match — compared via `ct_eq` (invariant I-5).
56//! assert!(bool::from(shared_secret.expose_secret().ct_eq(recovered_secret.expose_secret())));
57//! # Ok::<(), Box<dyn std::error::Error>>(())
58//! ```
59//!
60//! ## Mathematical Correctness
61//!
62//! The ML-KEM implementation is based on the Module-LWE (Learning with Errors)
63//! hardness problem, which provides provable quantum resistance. The security
64//! reduction ensures that breaking ML-KEM requires solving the Module-LWE problem,
65//! which is believed to be hard even for quantum computers.
66//!
