6//! # Digital Signatures
7//!
8//! This module provides a unified interface for both post-quantum and classical
9//! digital signature schemes. It supports the following algorithms:
10//!
11//! ## Post-Quantum Algorithms
12//!
13//! ### ML-DSA (FIPS 204)
14//! Module-Lattice-Based Digital Signature Algorithm providing EUF-CMA security.
15//!
16//! - **ML-DSA-44**: NIST Security Category 2 (~128-bit classical security)
17//!   - Public key: 1312 bytes
18//!   - Secret key: 2560 bytes
19//!   - Signature: ~2.4 KB
20//!
21//! - **ML-DSA-65**: NIST Security Category 3 (~192-bit classical security)
22//!   - Public key: 1952 bytes
23//!   - Secret key: 4032 bytes
24//!   - Signature: ~3.3 KB
25//!
26//! - **ML-DSA-87**: NIST Security Category 5 (~256-bit classical security)
27//!   - Public key: 2592 bytes
28//!   - Secret key: 4896 bytes
29//!   - Signature: ~4.6 KB
30//!
31//! ### SLH-DSA (FIPS 205)
32//! Stateless Hash-Based Digital Signature Algorithm providing EUF-CMA security
33//! with conservative security guarantees.
34//!
35//! - **SLH-DSA-SHAKE-128s**: NIST Security Category 1
36//!   - Very long keys, but quantum-resistant
37//!   - Stateless: No secret state that needs updating
38//!   - Signatures can be made as many times as needed
39//!
40//! - **SLH-DSA-SHAKE-192s**: NIST Security Category 3
41//!   - Balanced security and performance
42//!
43//! - **SLH-DSA-SHAKE-256s**: NIST Security Category 5
44//!   - Highest security level
45//!
46//! ### FN-DSA (FIPS 206)
47//! Few-Time Digital Signature Algorithm based on NTRU lattice.
48//! Provides compact signatures with lattice-based security.
49//!
50//! - **FN-DSA-512**: ~128-bit security (Level I)
51//!   - Public key: 897 bytes
52//!   - Secret key: 1281 bytes
53//!   - Signature: 666 bytes
54//!
55//! - **FN-DSA-1024**: ~256-bit security (Level V)
56//!   - Public key: 1793 bytes
57//!   - Secret key: 2305 bytes
58//!   - Signature: 1280 bytes
59//!
60//! ## Classical Algorithms
61//!
62//! - **ECDSA**: Elliptic Curve Digital Signature Algorithm
63//!   - P-256, P-384, P-521 curves
64//!
65//! - **Ed25519**: EdDSA signature scheme using Curve25519
66//!   - Fast, secure, widely adopted
67//!
68//! ## Security Properties
69//!
70//! All signature implementations in this module provide:
71//!
72//! - **EUF-CMA**: Existential Unforgeability under Chosen Message Attacks
73//!   - An adversary cannot forge a valid signature even when given
74//!     signatures on arbitrary messages of their choice
75//!
76//! - **Constant-time**: All secret-handling operations execute in constant time
77//!   - Protects against timing side-channel attacks
78//!
79//! - **Zeroization**: Secret keys are securely wiped when dropped
80//!   - Prevents secret material from remaining in memory
81//!
82//! - **Domain Separation**: Optional context string parameter
83//!   - Prevents signature reuse across different applications
84//!
85//! ## Algorithm Comparison
86//!
87//! | Algorithm | Key Size | Signature Size | Security | Best For |
88//! |-----------|-----------|----------------|----------|----------|
89//! | ML-DSA-44 | ~2.4 KB | ~2.4 KB | 128-bit | Balanced security/performance |
90//! | ML-DSA-65 | ~3.6 KB | ~3.3 KB | 192-bit | High-security applications |
91//! | ML-DSA-87 | ~4.6 KB | ~4.6 KB | 256-bit | Maximum security |
92//! | SLH-DSA-SHAKE | Large | ~41 KB | Conservative | Long-term security, stateless |
93//! | FN-DSA-512 | ~1.3 KB | 666 bytes | 128-bit | Compact signatures, fast verify |
94//! | FN-DSA-1024 | ~2.3 KB | 1280 bytes | 256-bit | High-security with compact sigs |
95//!
96//! ## Example Usage
97//!
98//! ### ML-DSA Signing and Verification
99//!
100//! ```no_run
101//! use arc_primitives::sig::{MlDsaParameterSet, generate_keypair, sign, verify};
102//!
103//! // Generate keypair
104//! let (pk, sk) = generate_keypair(MlDsaParameterSet::MLDSA65)?;
105//!
106//! // Sign a message
107//! let message = b"Hello, post-quantum world!";
108//! let signature = sign(&sk, message, &[])?;
109//!
110//! // Verify signature
111//! let is_valid = verify(&pk, message, &signature, &[])?;
112//! assert!(is_valid);
113//! # Ok::<(), Box<dyn std::error::Error>>(())
114//! ```
115//!
116//! ### SLH-DSA with Context
117//!
118//! ```no_run
119//! use arc_primitives::sig::slh_dsa::{SecurityLevel, SigningKey, VerifyingKey};
120//!
121//! let (sk, pk) = SigningKey::generate(SecurityLevel::Shake128s)?;
122//!
123//! let message = b"Important document";
124//! let context = b"my-application-v1"; // Domain separation
125//!
126//! let signature = sk.sign(message, Some(context))?;
127//! let is_valid = pk.verify(message, &signature, Some(context))?;
128//! # Ok::<(), Box<dyn std::error::Error>>(())
129//! ```
130//!
131//! ## FIPS Compliance
132//!
133//! All post-quantum signature algorithms are fully compliant with their respective FIPS standards:
134//!
135//! - **ML-DSA**: [FIPS 204](https://csrc.nist.gov/pubs/fips/204/final/)
136//! - **SLH-DSA**: [FIPS 205](https://csrc.nist.gov/pubs/fips/205/final/)
137//! - **FN-DSA**: [FIPS 206](https://csrc.nist.gov/pubs/fips/206/final/)
138//!
139//! All implementations use audited FIPS-standard crates (fips204, fips205, fn-dsa).
140//!
141//! ## Mathematical Correctness
142//!
143//! ### ML-DSA
144//! Based on Module-SIS (Short Integer Solution) lattice problem.
145//! The security reduction proves that forging a signature requires solving
146//! the Module-SIS problem, which is believed to be hard for quantum computers.
147//!
148//! ### SLH-DSA
149//! Based on Merkle tree hash-based signatures. Security relies on
150//! collision resistance of the underlying hash function (SHAKE).
151//! Provides information-theoretic security against quantum attacks.
152//!
153//! ### FN-DSA
154//! Based on the NTRU lattice problem (Shortest Vector Problem).
155//! Provides provable security with compact signatures.
156//!
157//! ## Security Considerations
158//!
159//! - **Key Generation**: Always use cryptographically secure random number generators
160//! - **Key Storage**: Store secret keys securely, never transmit them
161//! - **Signature Reuse**: SLH-DSA is stateless (no signature reuse concerns)
162//! - **Context Parameter**: Use non-empty context strings for domain separation
163//! - **Algorithm Selection**: Choose algorithm based on your security requirements
164//!   and performance constraints
165//!

169//! - [`slh_dsa`]: SLH-DSA (FIPS 205) hash-based signatures
