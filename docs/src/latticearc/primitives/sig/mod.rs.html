6//! # Digital Signatures
7//!
8//! This module provides a unified interface for both post-quantum and classical
9//! digital signature schemes. It supports the following algorithms:
10//!
11//! ## Post-Quantum Algorithms
12//!
13//! ### ML-DSA (FIPS 204)
14//! Module-Lattice-Based Digital Signature Algorithm providing EUF-CMA security.
15//!
16//! - **ML-DSA-44**: NIST Security Category 2 (~128-bit classical security)
17//!   - Public key: 1312 bytes
18//!   - Secret key: 2560 bytes
19//!   - Signature: ~2.4 KB
20//!
21//! - **ML-DSA-65**: NIST Security Category 3 (~192-bit classical security)
22//!   - Public key: 1952 bytes
23//!   - Secret key: 4032 bytes
24//!   - Signature: ~3.3 KB
25//!
26//! - **ML-DSA-87**: NIST Security Category 5 (~256-bit classical security)
27//!   - Public key: 2592 bytes
28//!   - Secret key: 4896 bytes
29//!   - Signature: ~4.6 KB
30//!
31//! ### SLH-DSA (FIPS 205)
32//! Stateless Hash-Based Digital Signature Algorithm providing EUF-CMA security
33//! with conservative security guarantees.
34//!
35//! - **SLH-DSA-SHAKE-128s**: NIST Security Category 1
36//!   - Very long keys, but quantum-resistant
37//!   - Stateless: No secret state that needs updating
38//!   - Signatures can be made as many times as needed
39//!
40//! - **SLH-DSA-SHAKE-192s**: NIST Security Category 3
41//!   - Balanced security and performance
42//!
43//! - **SLH-DSA-SHAKE-256s**: NIST Security Category 5
44//!   - Highest security level
45//!
46//! ### FN-DSA (draft FIPS 206 / Falcon)
47//! Fast-Fourier Lattice-based signature scheme based on NTRU lattices. NIST
48//! selected Falcon for standardization as FIPS 206, but **the final standard
49//! has not been published yet** (Falcon remains in the standardization
50//! pipeline as of this writing).
51//! "FN" stands for "Falcon-NTRU"; despite the name, FN-DSA is a *many-time*
52//! signature scheme — there is no upper bound on the number of signatures
53//! producible per key (prior revisions of this doc incorrectly called it
54//! "Few-Time"). Provides compact signatures with lattice-based security.
55//!
56//! - **FN-DSA-512**: NIST Category 1 (~103-bit quantum, ~128-bit classical)
57//!   - Public key: 897 bytes
58//!   - Secret key: 1281 bytes
59//!   - Signature: ~666 bytes average (variable-length encoding)
60//!
61//! - **FN-DSA-1024**: NIST Category 5 (~230-bit quantum, ~256-bit classical)
62//!   - Public key: 1793 bytes
63//!   - Secret key: 2305 bytes
64//!   - Signature: ~1280 bytes average
65//!
66//! ## Classical Algorithms
67//!
68//! This crate only exposes **Ed25519** (EdDSA over Curve25519) as a classical
69//! signature primitive. It is used in the hybrid ML-DSA-65 + Ed25519
70//! construction in [`hybrid::sig_hybrid`](crate::hybrid::sig_hybrid) and as a
71//! standalone signer via [`primitives::ec::ed25519`](crate::primitives::ec::ed25519).
72//! **ECDSA over NIST P-curves (P-256/P-384/P-521) is NOT implemented** —
73//! prior revisions of this doc listed them, which was aspirational. For
74//! secp256k1 blockchain / interoperability use cases, see
75//! `crate::primitives::ec::secp256k1` (available with the default feature set;
76//! gated out under `--features fips`).
77//!
78//! ## Security Properties
79//!
80//! All signature implementations in this module provide:
81//!
82//! - **EUF-CMA**: Existential Unforgeability under Chosen Message Attacks
83//!   - An adversary cannot forge a valid signature even when given
84//!     signatures on arbitrary messages of their choice
85//!
86//! - **Constant-time**: All secret-handling operations execute in constant time
87//!   - Protects against timing side-channel attacks
88//!
89//! - **Zeroization**: Secret keys are securely wiped when dropped
90//!   - Prevents secret material from remaining in memory
91//!
92//! - **Domain Separation**: Optional context string parameter
93//!   - Prevents signature reuse across different applications
94//!
95//! ## Algorithm Comparison
96//!
97//! | Algorithm | Key Size | Signature Size | Security | Best For |
98//! |-----------|-----------|----------------|----------|----------|
99//! | ML-DSA-44 | ~2.4 KB | ~2.4 KB | 128-bit | Balanced security/performance |
100//! | ML-DSA-65 | ~3.6 KB | ~3.3 KB | 192-bit | High-security applications |
101//! | ML-DSA-87 | ~4.6 KB | ~4.6 KB | 256-bit | Maximum security |
102//! | SLH-DSA-SHAKE | Large | ~41 KB | Conservative | Long-term security, stateless |
103//! | FN-DSA-512 | ~1.3 KB | 666 bytes | 128-bit | Compact signatures, fast verify |
104//! | FN-DSA-1024 | ~2.3 KB | 1280 bytes | 256-bit | High-security with compact sigs |
105//!
106//! ## Example Usage
107//!
108//! ### ML-DSA Signing and Verification
109//!
110//! ```no_run
111//! use latticearc::primitives::sig::ml_dsa::{MlDsaParameterSet, generate_keypair};
112//!
113//! // Generate keypair
114//! let (pk, sk) = generate_keypair(MlDsaParameterSet::MlDsa65)?;
115//!
116//! // Sign a message (empty context = no domain separation)
117//! let message = b"Hello, post-quantum world!";
118//! let signature = sk.sign(message, &[])?;
119//!
120//! // Verify signature
121//! let is_valid = pk.verify(message, &signature, &[])?;
122//! assert!(is_valid);
123//! # Ok::<(), Box<dyn std::error::Error>>(())
124//! ```
125//!
126//! ### SLH-DSA with Context
127//!
128//! ```no_run
129//! use latticearc::primitives::sig::slh_dsa::{SlhDsaSecurityLevel, SigningKey, VerifyingKey};
130//!
131//! let (sk, pk) = SigningKey::generate(SlhDsaSecurityLevel::Shake128s)?;
132//!
133//! let message = b"Important document";
134//! let context = b"my-application-v1"; // Domain separation
135//!
136//! let signature = sk.sign(message, context)?;
137//! let is_valid = pk.verify(message, &signature, context)?;
138//! # Ok::<(), Box<dyn std::error::Error>>(())
139//! ```
140//!
141//! ## FIPS Compliance
142//!
143//! All post-quantum signature algorithms are fully compliant with their respective FIPS standards:
144//!
145//! - **ML-DSA**: [FIPS 204](https://csrc.nist.gov/pubs/fips/204/final/)
146//! - **SLH-DSA**: [FIPS 205](https://csrc.nist.gov/pubs/fips/205/final/)
147//! - **FN-DSA**: [FIPS 206 (pending, Falcon selected for standardization)](https://csrc.nist.gov/Projects/post-quantum-cryptography)
148//!
149//! All implementations use audited FIPS-standard crates (fips204, fips205, fn-dsa).
150//!
151//! ## Mathematical Correctness
152//!
153//! ### ML-DSA
154//! Based on Module-SIS (Short Integer Solution) lattice problem.
155//! The security reduction proves that forging a signature requires solving
156//! the Module-SIS problem, which is believed to be hard for quantum computers.
157//!
158//! ### SLH-DSA
159//! Based on Merkle tree hash-based signatures. Security relies on
160//! collision resistance of the underlying hash function (SHAKE).
161//! Provides information-theoretic security against quantum attacks.
162//!
163//! ### FN-DSA
164//! Based on the NTRU lattice problem (Shortest Vector Problem).
165//! Provides provable security with compact signatures.
166//!
167//! ## Security Considerations
168//!
169//! - **Key Generation**: Always use cryptographically secure random number generators
170//! - **Key Storage**: Store secret keys securely, never transmit them
171//! - **Signature Reuse**: SLH-DSA is stateless (no signature reuse concerns)
172//! - **Context Parameter**: Use non-empty context strings for domain separation
173//! - **Algorithm Selection**: Choose algorithm based on your security requirements
174//!   and performance constraints
175//!
176//! ## Module Structure
177//!
178//! - [`ml_dsa`]: ML-DSA (FIPS 204) lattice-based signatures
179//! - [`slh_dsa`]: SLH-DSA (FIPS 205) hash-based signatures
180//! - [`fndsa`]: FN-DSA (FIPS 206) compact lattice signatures
181
